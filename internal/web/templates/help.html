Triage legend

Each comment in the corpus is one of Steven's past reviews. Triage tags it for what kind of signal it carries, so the persona compiler can weight or route it correctly. Triage does not immediately write to the vault — that's a separate bulk step (ingest synthesize) so we can review what we're about to land before committing.

keydecisionmeaning
h hard Blocking rule. The bot must always flag this pattern. Example: "should check capability not role."
s soft Preference. Bot mentions if relevant but doesn't block. Example: "prefer X over Y here."
p personal My taste; I won't push back if the author disagrees. Bot suggests gently or stays silent.
t tradeoff Explains why something is the way it is. Documents rationale; doesn't enforce a rule. Feeds tradeoffs/.
c style Concrete code-pattern example (good or bad) for the style/ tree.
r praise ✓ Good pattern. Voice sample for "how Steven praises" + positive example.
k skip Noise. LGTM, "thanks", merge-conflict chatter, customer support. Not training data.
n needs-thought Come back to this. Doesn't get synthesized.

Other keys

jnext pending comment (flat-list view)
kprevious pending comment (flat-list view, conflicts with skip on the PR-detail page)
oopen the comment on github

Synthesis lifecycle

  1. pending → you triage with one of the 8 keys above.
  2. Triage sets status, decision, and an optional note. Nothing else happens yet.
  3. Later (whenever you're done triaging a batch), ingest synthesize reads everything with a decision set and synthesized=0, writes the appropriate vault entries in my-agent under agent/vault/projects/steven-reviewer/, and flips synthesized=1.
  4. Synthesis is idempotent; rerunning processes only the not-yet-synthesized rows.
